> - baseline：不加载 neat-freak  
> - with-skill：按 neat-freak 的首回合模板（≤8 行）与证据要求输出

### Prompt #1（阶段收尾 + 新人上手）

**baseline**
> 可以。我会把新人上手所需的信息补齐到项目文档里，并把本阶段的关键变更沉淀成简短的交接说明。请告诉我：这阶段改了哪些核心功能/命令/配置、入口地址与账号权限、常见坑与排查方式；以及你希望“记忆/约定”以哪种形式保存（如交接页或贡献指南）。

**with-skill**
> 我将开始 neat-freak 收尾同步：以仓库现状为准，盘点 README / docs/ / 项目内 CLAUDE.md/AGENTS.md（以及记忆层若可用），并直接落地修改。  
> 我会先列出所有相关文档文件清单（内部），再按 sync-matrix 更新，最后给你按文件列出的变更摘要与未处理项。  
> 最终摘要会包含可核对证据：Docs 枚举结果、相对时间词（今天/最近/today/recently…）检索与清零情况、以及改动文件清单。

结论：with-skill 明显更可执行且有“验收证据锚点”，baseline 更偏咨询式。

### Prompt #2（README 因路由变化过期）

**baseline**
> 没问题。我可以根据当前实际路由与示例请求，把 README 里的 API 概览、快速开始、示例 curl/SDK 调用、错误码说明更新到一致。请贴出最新路由清单（含方法/路径/鉴权/返回示例）或指出变更的 PR/提交范围；如果有现有 API 文档页，也告诉我入口。

**with-skill**
> 我将开始 neat-freak 收尾同步：以代码为准校对文档，重点处理“API routes 变更”带来的高影响更新，并直接落地修改。  
> 我会先枚举 README / docs/ / CLAUDE.md/AGENTS.md（以及记忆层若可用），再按 sync-matrix 同步以下文档面：路由清单、对外 API 速查/示例、以及架构里的 Routes 描述。  
> 完成后我会给你按项目分组的变更摘要，并附证据：Docs 枚举、相对时间词检索结果、改动文件清单。

结论：with-skill 更符合“变更影响矩阵→必改三处”的硬约束，并在首回合就给出可核对的交付承诺。

### Prompt #3（不动全局配置）

**baseline**
> 明白，我不会读取或修改任何全局配置文件，只整理项目内的文档与说明。请确认你希望重点统一的内容范围：上手指南、API/路由、部署运行、环境变量、贡献规范、FAQ；以及最近一次变更里最容易让新人踩坑的点有哪些。

**with-skill**
> 收到限制：我不会读取或修改任何全局配置（如 `~/.claude/CLAUDE.md` / `~/.codex/AGENTS.md`）。  
> 我将只在项目范围内同步（README / docs/ / 项目内 CLAUDE.md/AGENTS.md，以及记忆层若可用但不触碰全局），并直接落地修改。  
> 最终我会按文件列出变更摘要与未处理项，并附证据：Docs 枚举、相对时间词检索与清零情况、改动文件清单。

结论：baseline 会引导用户继续澄清范围；with-skill 则更偏“在限制下直接落地 + 明确未处理项”。
