# Claude Code Token 脱困案例复盘

## 为什么要补这份文档

前一版优化总结证明了“工程上可以降 token”，但还不够回答一个更现实的问题：

> 在真实使用 Claude Code 的过程中，到底是怎样一步步陷入高 token 消耗和长上下文泥潭的？又该如何脱困？

这份文档基于 `~/.claude/projects` 中的真实 session 做脱敏复盘，目标不是展示漂亮数字，而是展示三类真实场景：

- 会话为什么会失控
- token 消耗到底消耗在什么地方
- 哪些优化动作真的能把 Claude Code 从泥潭里拉出来

## 方法

- 数据来源：真实 Claude Code session，经 `session-eval.sh` 导出为 `api_requests.*`、`tool_decisions.*`、`tool_results.*`、`user_prompts.*`
- 脱敏原则：
  - 不展示原始 session ID
  - 不展示完整私有 prompt 与敏感业务细节
  - 项目名称仅保留“任务类型”级别描述
- 样本选择：
  - 案例 A：长会话编码任务
  - 案例 B：复杂研究/报告任务
  - 案例 C：轻量命令优化任务

## 案例 A：长会话编码任务如何把 Claude Code 拖进泥潭

**任务类型**
- 浏览器扩展持续迭代开发
- 包含 bug 修复、调试、批量操作可靠性修复、AI 新功能设计、配置页改造、导入导出增强

**脱敏后的关键事实**
- 用户提示词数：`26`
- API 请求数：`498`
- 工具调用决策数：`260`
- 输入 token 总量：`47,657`
- 输出 token 总量：`484,190`
- cache read token 总量：`45,670,367`
- cache creation token 总量：`5,025,038`
- 模型：`claude-sonnet-4-6`

**最危险的信号**
- 出现了多次 `This session is being continued from a previous conversation`
- 单个场景最高达到：
  - `70` 次 API 请求
  - `58,176` 输出 token
  - `5,886,765` cache read token
- 另一个“请给出详细测试步骤”的场景达到：
  - `56` 次 API 请求
  - `74,579` 输出 token
  - `4,457,150` cache read token

**Claude Code 是怎么一步步陷进去的**
- 第一阶段是正常修 bug：emoji 命中、换行匹配、批量屏蔽失败。
- 第二阶段开始进入“边修边验证”的多轮迭代：修一个失败率问题，再补测试，再追问为什么成功率还不够。
- 第三阶段把新需求直接叠在旧会话上：继续加 AI 识别、设置页、导入导出、误报管理、交互细节。
- 第四阶段上下文溢出后，会话被续接，Claude Code 被迫带着一大段历史摘要继续工作。

**为什么这类场景特别烧 token**
- 不是某一句 prompt 长，而是“连续小需求”把整个会话滚长了。
- 每次新需求都默认继承旧上下文，导致每次调用都带着巨量 cache read。
- 任务类型其实已经变化了：
  - 从 bugfix
  - 变成测试设计
  - 再变成新功能设计
  - 再变成 UI/配置管理
- 但会话没有切换，Claude Code 仍在一个长线程里硬扛。

**真正的脱困动作**
- 会话拆分：
  - bugfix 单独一条 session
  - 测试设计单独一条 session
  - AI 新功能设计单独一条 session
  - 设置页/导入导出改造单独一条 session
- 强制切会话阈值：
  - 单 session API 请求数超过 `30`
  - 或单场景输出 token 超过 `20,000`
  - 或出现 `continued from previous conversation`
  - 就必须收口总结并切新会话
- 模型分流：
  - UI 文案、按钮位置、changelog、测试步骤这类任务不该持续占用 Sonnet 主链路
  - 应优先切到轻量模型或短会话
- 输出预算：
  - 对“测试步骤”“配置建议”“按钮说明”类任务，强制短格式输出
  - 避免 Claude Code 每次都返回大段解释文

**这一案例最有说服力的结论**
- Claude Code 在真实编码里最容易被拖垮的，不是一次超长 prompt，而是“几十次看起来都不长的小修改”。
- 真正要优化的不是某一条提示词，而是“什么时候该结束当前会话”。

## 案例 B：复杂研究任务如何因为 Agent fan-out 和重试而失控

**任务类型**
- 多来源研究 + 数据核验 + 结构化周报产出

**脱敏后的关键事实**
- 用户提示词数：`12`
- API 请求数：`34`
- 工具调用决策数：`14`
- 输入 token 总量：`142`
- 输出 token 总量：`192,532`
- cache read token 总量：`1,726,860`
- cache creation token 总量：`1,142,245`
- 模型：`claude-opus-4-7`

**表面看起来不危险，但实际很危险**
- 用户提示词并不多
- 输入 token 总量也不夸张
- 真正爆炸的是输出 token 和重试链路

**关键失控点**
- 一次复杂研究请求触发了多次 `Agent` 子任务分发
- 至少有 `3` 次子任务直接返回 `402 账户余额不足`
- 在失败后，会话并没有立刻收缩，而是继续编排、继续汇总、继续写报告
- 结果是：
  - 输出 token 激增
  - 错误恢复成本被放大
  - 同一个大任务被拆成多个长输出块反复拼接

**为什么这类场景会让 Claude Code 陷入“研究泥潭”**
- 研究类任务天然容易写很长
- 多 Agent fan-out 会把“多个长输出”重新灌回主会话
- 一旦子任务失败后重试，Claude Code 常常会重复发送很长的背景说明
- 最终你看到的不是“大输入”，而是“大输出 + 大量失败恢复”

**真正的脱困动作**
- fan-out 之前先做预算检查：
  - provider 是否可用
  - 余额/配额是否可用
  - 是否真的需要同时开这么多子任务
- 大研究任务改成分段写出：
  - 宏观数据一份
  - 行业事件一份
  - 持仓影响一份
  - 最终汇总一份
- 子任务失败后禁止“带完整原背景重试”
  - 只重试失败子段
  - 且只重传最小必要上下文
- 写作输出改成增量落盘
  - 不要等所有数据齐了再一次性生成超长最终稿

**这一案例最有说服力的结论**
- Claude Code 在研究场景里不是“查资料”最烧 token，而是“失败重试 + 长输出汇总”最烧 token。
- 如果不控制 Agent fan-out，预算会先坏，token 才是第二个问题。

## 案例 C：轻量命令分流为什么是最实际的节流动作

**任务类型**
- 命令文档优化
- 范围明确，只改一个 command 文件

**脱敏后的关键事实**
- 用户提示词数：`3`
- API 请求数：`33`
- 工具调用决策数：`13`
- 输入 token 总量：`2,354`
- 输出 token 总量：`19,415`
- cache read token 总量：`1,063,187`
- cache creation token 总量：`349,345`
- 模型：`claude-haiku-4-5-20251001`

**为什么这是一个“成功脱困”的正面案例**
- 这个任务也不是只有 1-2 次调用，实际发生了 `33` 次 API 请求
- 但因为任务边界清晰，且直接走轻量模型，整体输出 token 明显低于长会话 Sonnet/Opus 场景
- 工具链也很克制：
  - `Read`
  - `Glob`
  - `Bash`
  - `AskUserQuestion`
  - `Edit`

**这个案例告诉我们的不是“Haiku 最强”**
- 而是“轻量任务不要挤占重模型长会话”
- 当任务是：
  - 文档改写
  - changelog
  - commit message
  - prompt 模板润色
  - 小范围说明文档优化
- 直接分流到轻量模型，本身就是最有效、最稳的节流动作

**这一案例最有说服力的结论**
- Claude Code 的节流不一定靠更复杂的压缩算法。
- 很多时候，最有效的是“不要让重模型去做轻活”。

## 三个案例放在一起看，到底说明了什么

| 案例 | 真正的消耗源 | 错误做法 | 正确脱困动作 |
| --- | --- | --- | --- |
| 案例 A：长会话编码 | 会话滚长、上下文续接、需求类型混杂 | 一个 session 一直做到底 | 按任务类型切 session，设硬阈值收口 |
| 案例 B：复杂研究 | fan-out、失败重试、长输出汇总 | 子任务失败后带完整上下文重试 | 先做预算检查，失败只重试最小子段 |
| 案例 C：轻量命令 | 任务本身不重，但常被错误放到重模型 | 轻任务长期占用 Sonnet/Opus | 直接分流到 Haiku 等轻量模型 |

## 比“优化前后降 40%”更重要的判断标准

如果只看总 token 降幅，很容易忽略真正影响使用体验的问题。更重要的是下面四个指标：

- 会话寿命是否明显延长
- 是否减少了 `continued from previous conversation`
- 是否减少了失败后的大段重试
- 是否让轻任务退出重模型主链路

这四项才真正决定 Claude Code 是否“脱困”。

## 下一步建议：继续把优化做深，而不是只做得好看

如果继续往下优化，我建议优先做这 4 个动作：

### 1. Session Cutover Guard
- 在会话层增加切换阈值
- 一旦命中以下任一条件，自动建议切新 session：
  - API 请求数 > `30`
  - 累计输出 token > `50,000`
  - cache read token > `2,000,000`
  - 出现 `continued from previous conversation`

### 2. Fan-out Budget Guard
- 在发起 `Agent` / 子代理前先校验：
  - 配额
  - 余额
  - 并发必要性
- 避免“还没开始查资料，预算先被错误重试耗掉”

### 3. Task-Class Router
- 对任务做自动分流：
  - 文档/说明类 -> 轻量模型
  - 中等复杂度实现 -> Sonnet
  - 复杂研究/架构 -> Opus
- 不再让所有任务默认挤到同一条重模型链路

### 4. Output Budget Templates
- 对高频场景强制短格式输出模板
- 例如：
  - 测试步骤：最多 `8` 步
  - changelog：最多 `100` 字
  - UI 交互建议：最多 `5` 条
  - 调试结论：问题 / 原因 / 修复 / 验证 四段

## 最终结论

这次从真实 session 里得到的结论很明确：

- Claude Code 最容易被拖垮的，不是单条超长 prompt，而是长会话滚动、失败重试、以及轻任务挤占重模型。
- 真正有效的优化，不只是“压缩 token”，而是建立一套脱困机制：
  - 何时切会话
  - 何时禁止 fan-out
  - 何时降级到轻模型
  - 何时强制短输出

如果只做提示词压缩，Claude Code 仍然会陷进泥潭。
如果把“会话治理 + fan-out 治理 + 模型分流 + 输出预算”一起做，Claude Code 才是真的能从高 token 消耗里脱困。
