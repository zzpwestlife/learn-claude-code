## 评分理由（简述）

### 优势（高分项）
- **D2/D5（9/10）**：流程分阶段且可执行；“盘点 → 影响矩阵 → 实际修改 → 自检 → 摘要”闭环清晰，落地性强。
- **D6（9/10）**：references 引用路径清晰，`agent-paths.md` 与 `sync-matrix.md` 对执行有直接支撑。
- **D7（8.8/10）**：新增了 **Reusable Interface (R)** 与 **反锚定**，让 skill 可组合、可复用、可自证，结构稳健。

### 短板（主要扣分点）
- **D1（7/10）**：frontmatter 的 description 明确了“做什么/何时触发（触发词）”，但仍缺少显式的 **Do not use when / 不适用场景** 与 1-2 条“典型输入输出例子”的短句版（目前更多是触发词列表）。
- **D4（6.5/10）**：虽声明“仅矛盾需要用户介入”，但对高风险动作（尤其 **删除文件/大范围重写文档结构**）缺少一个明确的“人类确认检查点”（例如：删除前 AskUserQuestion 确认）。
- **D8（8/10，推演）**：测试用例设计合理，但未做“带 skill vs baseline”对照实测；实际表现仍可能受工具可用性/仓库规模/上下文窗口影响。

## 下一轮优化建议（按 ROI 排序）

1. **补齐 D1：在 frontmatter.description 增加“不适用场景 + 降级策略”**  
   - 不适用示例：用户只是要回答问题/没有项目文件可读/仅要写一段文字（非同步任务）  
   - 降级策略：误触发时改为普通回答或改走 `writing-plans`/`executing-plans`/`code-review` 等

2. **补齐 D4：增加“高风险动作确认点”**  
   - 删除文件前、跨项目大范围改动前，强制 AskUserQuestion 二选一确认（继续/跳过）

3. **做一次 D8 对照实测（带 skill vs baseline）**  
   - 最少选 2 条 prompt：路由变更、env 变更  
   - 对比输出是否：更少漏改、摘要可核对、误触发更少
