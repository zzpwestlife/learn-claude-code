## 1) 文章中与“写 Skill”直接相关的关键结论（可落地）

### 1.1 Skill 的四要素（SoK 定义）：S = (C, π, T, R)
- **C（适用条件）**：不要只靠关键词；要能区分语义场景，避免误触发。
- **π（执行策略）**：可执行步骤 + 决策逻辑；比“正确但空洞”的原则更有效。
- **T（终止条件）**：什么时候停止/什么时候回退；缺 T 容易“越做越错”。
- **R（可复用接口）**：给别的 skill/流程可调用的入口（输出格式/产物/副作用说明）。

### 1.2 “详细（Detailed）> 全面（Comprehensive）”的反直觉
文章引用的评测结果指出：面面俱到的 Comprehensive 反而可能降低通过率。典型失败模式：
- **样例/模板锚定**：模型盲从过时示例 → 编出不存在字段 → 引入无关资源。
- **覆盖过多边界**：导致注意力稀释、路由混乱、执行路径不收敛。

### 1.3 Skill 的内在上限（对写法的约束）
文章提到两类边界：
- **表达边界**：适合写成语言规则/流程的知识（IF-THEN、检查表）；不适合写“架构品味/直觉裁决（utility）”。
- **选择边界**：skill 太多（语义接近）会降低选择准确率；暗示要控制粒度与可组合数量（2-3 个协同最佳）。

### 1.4 两类有效内容：L1 确定性注入 vs L2 扩散激活
可对应到 skill 内容结构：
- **L1（确定性注入）**：明确条件→明确动作（IF-THEN），如“若 CPU>80% 且 5min 则扩容”。这类最可靠。
- **L2（扩散激活/风格指令）**：如“说人话/短句/生活比喻”。有效但分辨率有限，必须：
  - 避免矛盾（“说人话” vs “保持专业感”）
  - 避免冗余（“简洁直接” vs “多用短句”）
  - 尽量给少量高质量范例（“像这样写”优于抽象口号）

## 2) 把结论翻译成“改 Skill 的检查清单”

### A. 触发/路由（C）
- [ ] 明确 **误触发降级**：触发条件不满足时，STOP 并路由到更合适的 skill（或 AskUserQuestion）。
- [ ] 把“语义触发”写成可执行的 triage：1-3 个判断问题即可，不写成大段抽象描述。
- [ ] 避免“语义近邻冲突”：若两个 skill 很像，明确边界与优先级。

### B. 执行策略（π）
- [ ] 把策略写成**最短可复现路径**（MVP），优先覆盖“高频成功路径”，而不是穷尽边界。
- [ ] 每一步都要有“产物/中间态”（比如文件、命令输出、diff、日志片段），可被下游复用。
- [ ] 对“模板/示例”加防锚定声明：示例仅作参考，必须以当前环境/真实接口/真实代码为准。

### C. 终止条件（T）
- [ ] 明确 STOP 条件（缺输入/缺验证命令/无法复现/无 diff 等），防止“继续瞎做”。
- [ ] 明确验证门（能跑测试则必须跑；不能跑则要求替代证据）。
- [ ] 明确回退策略（无法继续 → 需要用户补哪些信息）。

### D. 可复用接口（R）
- [ ] 输出格式固定（例如：计划模板、证据块模板、审查报告结构）。
- [ ] 声明副作用（是否写文件、是否会改 repo、是否会创建分支）。

### E. L1 / L2 内容配比
- [ ] L1（规则/流程）放在最前面（高确定性、高复用）。
- [ ] L2（风格/偏置）控制数量，避免矛盾/冗余；用少量范例提高分辨率。

## 3) 建议的落地顺序（先从工程影响面最大处改）

1. `verification-before-completion`：把“证据块”模板作为 R；把 STOP/替代证据优先级写死（T）。
2. `executing-plans`：强制“计划存在 + 每步验证命令存在”才进入执行（C/T）；每步产物可复用（π/R）。
3. `test-driven-development`：明确 RED/GREEN 证据字段（R）；缺测试命令则 STOP（T）。
4. `code-review`：无 diff/无目标则 STOP（C/T）；示例避免锚定（π）。
5. `brainstorming` / `writing-plans` / `design-first`：减少“全面覆盖”，增强 triage/降级与范例（C/π）。
