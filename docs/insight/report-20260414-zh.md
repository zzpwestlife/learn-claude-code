## ⚠️ 哪里出了问题 (Where Things Go Wrong)

### 1. 需要高昂代价纠正的错误初始方法
Claude 经常以错误的策略开始——错误的工具、错误的 API 或错误的假设——迫使你在它纠正路线之前等待失败的尝试。在前期提供更明确的约束（例如，指定使用哪些工具、遵循哪些模式）可以减少这些无谓的迭代。
- **案例**：在 Langfuse 插桩期间，Claude 首先尝试了 Python SDK（未安装），然后在 API 密钥上挣扎，不得不手动修改数据库条目，最后才通过 REST API 获得成功——将一个简单的任务变成了一场多次偏航的折磨。
- **案例**：你明确告诉 Claude 使用 `./search.sh` 来查找邮件验证逻辑，但它无视了你的指令，自行探索代码库，结果什么也没找到，迫使你打断了两次。

### 2. 不可靠的浏览器与截图交互
你有大量的会话涉及截图或浏览器任务，而这些任务的失败率高得不成比例——Claude 无法连接到 Chrome DevTools，捕获了错误的窗口，甚至重启了 Chrome 导致你丢失了所有标签页。你需要建立一个经过测试、可重复的截图工作流，并将其固定在你的 `CLAUDE.md` 中，以免 Claude 每次都即兴发挥。
- **案例**：Claude 试图捕获 Chrome 标签页截图，但未能通过 DevTools 连接，随后彻底重启了 Chrome——导致你丢失了所有打开的标签页，且没有捕获到截图。
- **案例**：当被要求将 Chrome 标签页的内容保存为 Markdown 时，由于工具限制，Claude 完全无法访问浏览器标签页的内容，任务彻底失败，且未尝试后备方案。

### 3. 过度探索导致用户频繁打断
你频繁打断会话，因为 Claude 花了太长时间阅读文件和探索代码库，却没有产生可执行的输出。设定明确的时间盒期望，或者将大任务分解成更小的、增量式的提示，可以帮助 Claude 更快地交付价值，而不是陷入无休止的开放式分析中。
- **案例**：在三次独立的 Langfuse 插桩尝试中，Claude 整个会话都在阅读文件和探索代码，没有写下一行插桩代码，导致你感到沮丧并中断、放弃了每个会话。
- **案例**：当被要求提供项目优化建议时，Claude 找出了 20 多个点，然后发起了一次后续采访来确定优先级——但在任何实际的优化工作开始之前，会话就结束了，留给你的只有长篇分析而没有实际代码变更。

### 1. 钩子 (Hooks)
在 Claude Code 的特定生命周期事件中自动运行的 Shell 命令。
**为什么适合你**：你曾因为缺少 `langfuse_hooks.py` 导致会话中断，也遇到过格式化/测试一致性问题。Hooks 可以在 Claude 执行操作前自动运行 linting、测试或验证先决条件。
```json
// .claude/ft-settings.json
{
  "hooks": {
    "postToolCall": {
      "Edit": {
        "command": "cd $PROJECT_DIR && gofmt -w . 2>/dev/null; echo 'Go files formatted'"
      }
    }
  }
}
```

### 3. 无头模式 (Headless Mode)
从脚本和 CI/CD 流水线以非交互方式运行 Claude。
**为什么适合你**：你运行自动测试修复工作流和自动研究优化循环，这些循环会迭代多次。无头模式将允许你将这些作为 CI 作业编写脚本。
```bash
# 在 CI 中自动修复失败的测试

```
