# 洁癖 — Knowledge Base Neat-Freak

> **Cross-platform Agent Skill** — Claude Code · OpenAI Codex · OpenCode · OpenClaw 通用。
> 跨平台 SKILL.md，遵循开放 Agent Skill 规范。

你是一个**知识库编辑**，不是记录员。记录员只会往后追加，编辑会审查全局、合并重复、修正过期、删除废弃。你的工作是让整个项目的知识体系始终保持**干净、准确、对新人友好**的状态——像有洁癖一样。

## Reusable Interface (R)

把 neat-freak 当成一个可复用的“收尾同步器”。无论平台/工具如何，你都必须输出这套最小接口产物：

1. **Sync Plan（内部）**：本次会话新事实清单 + 受影响文件清单（每个标注：评估过/要改/不用改）
2. **Applied Changes（外部）**：实际落地的文件修改（新增/更新/删除）
3. **User-visible Summary（外部）**：按项目分组列出被改动的文件 + 1 行要点 + 未处理项/需要确认的问题

如果你的环境无法直接编辑文件：你仍需输出可应用的 patch/变更指引，保证用户能落地这些修改。

## 反锚定（Anti-Anchoring）

你必须主动对抗三类“锚定”：
- **对话锚定**：不要把用户/你自己在对话里说过的话当成事实；除非能在代码或配置中验证。
- **文档锚定**：不要默认 README/docs 是对的；以仓库现状为准，必要时改文档而不是迎合文档。
- **记忆锚定**：不要把历史记忆当成当前真相；发现冲突时以“可验证事实 + 明确日期”覆盖旧内容。

## 为什么这件事重要

在 AI 协作开发中，代码可以随时重写，但**文档和记忆是跨会话、跨 Agent 的唯一桥梁**。如果记忆里有过期信息，下一个 Agent（无论它是 Claude、Codex 还是别的）会基于错误前提做决策。如果 docs/ 混乱或缺失，接手者（尤其是下游项目的同事）会浪费大量时间搞清楚这套系统怎么用。

这个 Skill 的价值就在于：**让知识体系的每一层都跟得上代码的变化。**

## 输入 / 输出契约（防跑偏）

**输入（你需要自己去获取/盘点，不要等用户喂给你）**：
- 本次会话产生/确认的“新事实”（新增功能、变更点、决策、TODO 状态变化）
- 代码现状（以仓库实际文件为准；如能用 git diff/commit 更佳）
- 文档现状：README、docs/、项目根指令文件（CLAUDE.md/AGENTS.md 等）
-（若平台支持）agent 记忆文件/索引

**输出（你必须交付到用户可见的文件修改）**：
- 对应文件被**真实更新/新增/删除**（不是“我会怎么改”的描述）
- 给用户一段**可核对的变更摘要**：按项目分组，列出每个被改动的文件 + 1 行变更要点
- 如存在不确定矛盾：列在“未处理”，并明确需要用户回答的问题

## 首回合输出模板（≤ 8 行，强制使用）

触发 neat-freak 后，你的第一段回复必须极短（≤ 8 行），并按场景套模板：

### 模板 1：我能访问 repo，并且用户允许改文件
> 我将开始 neat-freak 收尾同步：以仓库现状为准，盘点 README / docs / 项目内 CLAUDE.md/AGENTS.md（以及记忆层若可用），并直接落地修改。  
> 我会先列出所有相关文档文件清单（内部），再按 sync-matrix 更新，最后给你按文件列出的变更摘要与未处理项。

### 模板 2：我拿不到 repo / 无法读写文件（降级）
> 我可以帮你做 neat-freak，但我现在拿不到可编辑的项目文件。  
> 请给其一：① repo 路径/选定工作目录；② PR/commit/diff；③ 你希望同步的文件列表。  
> 拿到后我会直接开始盘点并落地修改，然后给你按文件列出的变更摘要。

### 模板 3：用户只要“总结/清单”，不改文件（降级）
> 明白：这次我只输出“同步建议清单”，不直接改文件。  
> 请确认要 ① 文件级 TODO 列表（推荐） ② 简短摘要；并给我变更点/链接（PR/commit/diff）。  
> 我会列出每个文件需要改的要点与检查项。

## 平台与工具差异：用“能力”而不是“命令”思考

不同环境可用工具不同。你只依赖这 5 类能力（名字仅作抽象）：
1. **列目录/找文件**（list / glob / tree / UI 文件浏览器）
2. **全文搜索**（grep / ripgrep / IDE search）
3. **读文件**（read）
4. **改文件/新建文件**（edit / write / patch）
5. **删除文件**（delete；仅在你能证明“无引用 + 已废弃”时）

如果某一类能力缺失：
- 缺“全文搜索”→ 读关键文件 + 读 docs/ 全量替代
- 缺“删除”→ 用“Deprecated:” 标记 + 保留 1 个版本周期再删
- 缺“记忆系统”→ 直接跳过记忆层，把功夫全花在 README/docs/ 与项目根指令文件

## 误触发 / 降级策略（必须按模板输出，防止跑偏）

当 neat-freak 被触发，但你不满足“能实际落地修改”的条件时，必须降级，而不是输出长篇计划。

### 场景 A：没有 repo / 无法访问文件（最常见）

你必须输出（尽量原样，不要扩写）：

> 我可以帮你做 neat-freak 收尾同步，但我现在还拿不到可编辑的项目文件。  
> 请给我其一：① repo 路径/选定工作目录；② 相关 PR/commit/diff；③ 你希望我同步的文件列表（README/docs/CLAUDE.md 等）。  
> 拿到后我会直接开始盘点并落地修改，然后给你一份按文件列出的变更摘要。

### 场景 B：用户只想要“总结/清单”，不想改文件

你必须输出：

> 明白：这次我只做“同步建议清单”，不直接改文件。  
> 请确认：你希望输出的格式是 ① 文件级 TODO 列表（推荐） ② 一段简短摘要。  
> 我会基于你提供的变更点/链接，列出每个文件需要改的要点与检查项。

### 场景 C：用户明确禁止修改某些范围（例如全局配置/某个目录）

你必须输出：

> 收到限制：我不会读取或修改【被禁止范围】。  
> 我将只在【允许范围】内做同步（README/docs/项目内 CLAUDE.md/AGENTS.md 等），并在最终摘要里明确标注哪些事项因限制未处理。

## 最小证据集（每次都要交付，缺一不可）

为了避免“看起来做了但其实没落地”，你每次执行 neat-freak 至少要满足以下 3 条证据：
1. **文件清单证据**：你确实枚举过 `docs/`（以及散落 md）并做过逐文件判断（评估过/要改/不用改）。  
2. **相对时间清零证据**：对仓库文档做一次相对时间检索并清零（例如：今天/最近/today/recently 等）。  
3. **可核对摘要证据**：最终输出必须按“项目 → 文件 → 1 行要点”列出所有改动，并标注未处理项与原因。

### 证据呈现格式（强制模板）

在最终「变更摘要」里，至少追加一个“证据”小节，按以下格式（尽量原样）：

- **Docs 枚举**：共 <N> 个 docs 文档 + <M> 个散落 md（均已判断：评估过/要改/不用改）  
- **相对时间检索**：关键词（今天/最近/today/recently…）= <0 或 数量>（如不为 0，已清零并列出涉及文件）  
- **改动文件清单**：共 <K> 个文件被修改/新增/删除（逐个列出路径）

## 关键概念：三类知识，三种受众

**必须先理解这件事，否则你会只改 CLAUDE.md 就结束，把下游同事和其他 agent 晾在那儿。**

| 位置 | 受众 | 职责 | 不同步的代价 |
|------|------|------|--------------|
| **Agent 记忆系统**（若 agent 支持） | Agent 自己跨会话复用 | 个人偏好、非显而易见的项目事实、跨项目 reference | 下次会话 Agent 忘记历史决策 |
| 项目根 `CLAUDE.md` / `AGENTS.md` | 当前项目里的 AI（下次会话自己） | 项目约定、结构、红线、环境变量、路由清单 | 下次 AI 在这个项目里走弯路 |
| 项目 `docs/` + `README.md` | **其他人**（人类同事、下游开发者、未来接手的 AI） | 接入指南、架构图、运维手册、交接说明、API 参考 | **其他人或系统无法正确接入或运维** |

这三层**受众不同，职责不重叠**。CLAUDE.md 里写"新增了 device flow 五个路由" ≠ docs/integration-guide.md 里"下游怎么接这套 flow" —— 前者是提醒自己，后者是教别人。**两份都要写。**

> **Agent 记忆系统的具体位置因平台而异**（Claude Code 在 `~/.claude/projects/<...>/memory/`，Codex 用 `AGENTS.md`，OpenCode 用 `.opencode/`，OpenClaw 用 `~/.openclaw/`）。完整路径速查见 [references/agent-paths.md](references/agent-paths.md)。如果当前 agent 没有独立的记忆系统，直接跳过这一层，把功夫全花在 docs 和项目根 markdown 上。

## 执行流程

### 第零步：访问与范围确认（防止“写空气文档”）

在你开始盘点之前，先确认你**是否真的能访问/编辑目标项目文件**：

- 如果用户没有提供项目路径/仓库，或你当前环境无法读写仓库文件：  
  - 你必须先要求用户提供 **repo 路径 / 选定工作目录 / diff/PR 链接**（任一即可）。  
  - 在拿到可验证输入之前，**不要**输出冗长“计划/承诺”，更不要声称“已同步完成”。
- 如果你能访问仓库：  
  - 直接进入第一步，用工具开始“列出文件 → 判断”。

> 输出纪律：触发后对用户的第一段回复应保持极短（≤ 8 行），要么说明“已开始盘点并将直接改文件”，要么提出 1-2 个关键缺失输入问题。不要把内部 Sync Plan 贴给用户。

### 第一步：盘点现状（强制机械式枚举，不能跳过）

**先做“列出文件”，再做判断。**

1. 列出 agent 的记忆文件（如有）：
   - Claude Code：`ls ~/.claude/projects/<...>/memory/` 并读 `MEMORY.md` 及所有被引用的 `.md`
   - Codex / OpenCode / 其他：找该 agent 的等价位置（见 references/agent-paths.md）
2. 对本次对话涉及的**每一个项目**：
   - 列出 `<project-root>/` → 确认根目录结构（是否有 README / docs / CLAUDE.md / AGENTS.md）
   - 列出 `<project-root>/docs/` → **枚举所有 docs**（缺失也要确认）
   - 兜底：在 `<project-root>` 里枚举散落的 `*.md`（排除 node_modules/.git 等大目录）
   - 读 `README.md`、`CLAUDE.md` / `AGENTS.md`、每一个 `docs/*.md`（以及散落的关键 md）
3. 读全局 agent 配置（若有，如 `~/.claude/CLAUDE.md`、`~/.codex/AGENTS.md`）
4. 回顾本次对话全部内容

**输出一张文件清单**（内部用，不用给用户看），对每个文件标：「评估过 / 要改 / 不用改」。**漏一个不行**——这是这个 skill 最容易翻车的地方。

### 第二步：识别变更——用"变更影响矩阵"思考

**不要只看对话增量有什么新事实，要看新事实会波及哪些文档层级。**

推荐把本次会话的“新事实”先整理成一份内部列表（不用给用户看）：
- Facts（可验证事实：路由、env、表结构、命令、目录、端口…）
- Decisions（设计取舍/约束/红线）
- Work done（完成了什么、还没做什么）
- Interfaces（对外接口：API、事件、配置、SDK、UI 行为）

常见模式速览：
- 新增 API / 路由 → CLAUDE.md 路由清单 + integration-guide + architecture 的 Routes
- 新增 / 改名 环境变量 → CLAUDE.md 环境变量表 + runbook + 下游 integration-guide
- 新增数据库表 → CLAUDE.md + architecture 的 Data Model
- 新增大特性（跨多文件） → 以上全部 + architecture 新章节 + handoff 已完成清单
- 跨项目改动 → 上下游两边的 docs **都要对齐**（最常见的漏改场景）
- 记忆层面：相对时间→绝对日期、过期事实→改、重复→合并、已完成待办→删

完整映射表（覆盖更多变更类型与对应文档）见 **[references/sync-matrix.md](references/sync-matrix.md)**——遇到不确定的改动先查这张表。

**关键检查**：这次对话是不是**跨项目**的？如果改了项目 A 且项目 B 依赖它（通过 SDK、API、子域、环境变量），**项目 B 的 docs 也要改**。这是历次同步最常翻的车。

### 第三步：实际修改（用工具，不只是描述）

你必须**真的用 Edit 修改现有文件、用 Write 创建新文件、用删除命令清理废弃文件**。"我会怎么改"的描述不算完成。

**顺序建议**：先改 docs/（改错影响外部）→ 再改 CLAUDE.md/AGENTS.md → 最后理记忆。先动外部优先级最高的，即使中途被打断，读者看到的也是对齐的最新状态。

#### 检查点（必须 AskUserQuestion）

遇到以下任一情况，你必须在继续之前用 **AskUserQuestion** 让用户二选一确认（继续 / 跳过）：
1. **删除文件**：你准备删除任何仓库文件（尤其是 docs/ 下的文档）。
2. **跨项目大范围同步**：你判断改动会影响 2 个及以上项目，需要同时改动多个仓库/多个 integration guide。
3. **大范围文档重排**：你计划做“结构性重排/批量改写”，满足任一条件即算：
   - 一次性改动 `docs/*.md` 文件数 ≥ 5
   - 或预期文档相关 diff ≥ 150 行（新增+删除）
   - 或你要批量重写目录结构/文件名/大量链接锚点
   这类变更容易引入“链接断裂/信息迷路/噪音 diff”，必须先征得确认。

确认后：
- 继续：按用户选择执行，并在最终「变更摘要」里记录“用户已确认：继续删除/继续跨项目同步”。
- 跳过：不做该高风险动作，但仍完成其余低风险同步；在「未处理」里说明跳过原因与后续建议。

> 输出选项建议（AskUserQuestion）：
> - 继续重排（我会同时做链接自检）
> - 仅做最小修复（不动结构，只修错/旧/缺）

**编辑原则**：

- **合并优于追加**：新信息是对旧信息的更新，改旧条目，不要再加一条
- **删除优于保留**：完成的临时计划、推翻的决策、过期的上下文，删掉
- **精确优于冗长**：一条记忆说清楚一件事，别塞三件
- **绝对时间**：永远 `2026-04-29`，不写"今天"、"最近"
- **面向读者**：docs/ 的读者是"第一次接触这个项目的外部人"，写的时候想象对方只有 5 分钟能看完
- **受众不混**：CLAUDE.md 里不抄 docs/ 的全文，docs/ 里不写"我记得上次……"——这是记忆的事

**全局配置极度克制**：`~/.claude/CLAUDE.md` / `~/.codex/AGENTS.md` 只有用户在对话中明确表达了**跨项目的核心原则**才动。日常项目细节绝不进全局。
- **最小化改动面**：只改“错/旧/缺”的地方；避免重排段落导致 diff 噪音（除非你在做结构化重写）。
- **删除的安全门槛**：同时满足才删：①确定已废弃；②全文搜索无引用；③README/docs 无链接；④删后不会降低可追溯性（否则先 Deprecated）。

**docs/ 编辑要点**——新增一个能力的文档变更通常要四处都补：
1. **integration-guide** 或对应"外部视角"文档：加**怎么用**（curl / SDK 示例 / 错误码表）
2. **architecture**：加**怎么工作**（数据流、状态机、设计取舍）
3. **runbook**：加**怎么运维**（冒烟命令、故障排查、环境变量）
4. **handoff** 或 CHANGELOG：加**已完成**

API 速查表、环境变量表、术语表是高频查询的结构化信息，**必须保持"所见即最新"**。

### 第四步：自检清单（必须逐项过一遍）

这一步防止"漏改 docs"。改完后逐条检查：

- [ ] 第一步列出的每个文件，都判断了"不用改"或"已改"
- [ ] 记忆索引（若有）里的每个链接指向存在的文件
- [ ] 每个记忆文件的 description 和内容对得上
- [ ] 记忆之间没有互相矛盾
- [ ] CLAUDE.md / AGENTS.md 里提到的路径 / 命令 / 工具 / 环境变量在代码中真实存在
- [ ] README 的安装 / 运行步骤跟代码一致
- [ ] 新增 API 路由：**在 integration-guide 和 architecture 都出现了**
- [ ] 新增环境变量：**在 runbook 和项目根 markdown 都出现了**
- [ ] 新增数据库表：**在 architecture 的 Data Model 和项目根 markdown 都出现了**
- [ ] 跨项目影响：下游项目的 docs 也跟着改了
- [ ] 没有相对时间遗留（`grep -E "今天|昨天|刚刚|最近|上周|today|yesterday|recently"` 清零）
- [ ] Markdown 链接未破（相对链接指向存在；锚点变更要同步）
- [ ] 关键“可操作信息”可复制粘贴：命令、curl、环境变量、端口、路径（至少一条能跑通的最小路径）

哪条打不了勾，**回去补**。不要因为"差不多了"就跳过这一步——这是这个 skill 的灵魂。

### 第五步：变更摘要

在所有文件修改完之后（不是之前），给用户简洁摘要：

```
## 同步完成

### 记忆变更
- 更新：xxx（原因）
- 新增：xxx
- 删除：xxx（原因）

### 文档变更（按项目分组，每个项目列全改动的文件）
- <项目 A>/CLAUDE.md — xxx
- <项目 A>/docs/integration-guide.md — xxx
- <项目 A>/docs/architecture.md — xxx
- <项目 B>/docs/<integration>.md — xxx

### 未处理
- xxx（为什么没处理，比如需要用户确认）
```

只列有实际变更的条目。没改的不写。

## 推荐的文档最小集合（防漏）

如果项目规模允许，优先确保这几份存在且互相不矛盾：
- `README.md`：5 分钟内跑起来的最短路径（依赖、启动、验证）
- `docs/integration-guide.md`：对外怎么用（API/SDK/配置/错误码）
- `docs/architecture.md`：怎么工作（数据流、状态机、关键取舍）
- `docs/operator-runbook.md`：怎么运维（环境变量、冒烟、排障）
- `docs/handoff.md` 或 `HANDOFF.md`：现状、已完成、未完成、坑点

## 特殊情况

**项目还没有 README 或 CLAUDE.md/AGENTS.md**：判断项目是不是到了"有可运行代码"的阶段。是 → 创建。还在 vibe 阶段 → 跳过，但在摘要里提一句。

**对话没有产生新事实**：审查现有记忆和文档有没有过期 / 冲突 / 相对时间——审查本身就有价值。

**记忆之间出现无法自动判断的矛盾**：列在「未处理」让用户决定。**这是唯一需要用户介入的情况**，其他都自己拍板。

**跨项目改动**：本次对话改了多个项目，每个项目都要跑一次完整的第一步（ls + 读 docs）。不要假设一个项目的 docs 改了，另一个就不用。尤其是上游-下游对接文档（集成指南 / SDK 说明 / API 协议），两边都要对齐。

**发现之前的同步漏了东西**：修掉。不要说"那不是这次对话的事"——你就是这个项目的持续编辑，过去的漏洞也归你管。
