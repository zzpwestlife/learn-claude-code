# design-first (Skill)

架构/技术方案设计向导。需求或约束尚未澄清时，强制先完成设计并获得批准，再开始实施。适用于系统变更、新功能、技术选型等需要设计先行的场景。

**核心铁律**：没有代码、脚手架或伪代码，直到用户批准设计。

---

## 触发条件

需要做架构/技术方案（或变更系统行为）**且**需求/约束/风险尚未澄清。

```
帮我设计一个... / 我想加一个... / 怎么实现... / 技术方案
```

示例：
- ✅ "帮我设计一个用户认证系统，支持邮箱和 OAuth2 登录"
- ✅ "给 Express API 加缓存层，但还不确定用什么方案"

---

## 路由降级（不触发的情况）

| 情况 | 路由 |
|---|---|
| 已批准 spec → 写成实施计划 | `writing-plans` |
| 执行已有计划 | `executing-plans` |
| 审查代码改动 | `code-review` |
| 纯信息问答 / 改颜色等小修 | 直接处理 |

---

## 复杂度定级

| 级别 | 条件 | 流程深度 |
|---|---|---|
| **L 轻量** | ≤3 文件、无新依赖、无 schema/API 变更 | 1 问 → 1 段设计 → 批准摘要 |
| **M 中等** | 4-8 文件、或已有依赖新用法、或数据迁移 | ≤3 问 → 2-3 方案表 → Spec 文档 |
| **H 重度** | >8 文件、新系统/服务、新语言/运行时、多团队 | ≤5 问 → 4 角度攻击 → 逐段审批 |

---

## 六阶段流程

```
阶段1 探索定级
阶段2 深度追问（上游确定则跳过下游）
阶段3 方案探索与攻击测试（L 级跳过）
阶段4 分段设计与验证（逐段用户批准）
阶段5 写入 Spec 文档（L 级跳过）
阶段6 批准摘要 → 停止等待实施指令
```

---

## 交付物

1. **复杂度定级**：L / M / H + 理由
2. **方案对比表**：2-3 个方案（含最小化选项），含工作量/风险/回滚/依赖
3. **Spec 文档**（M/H 级）：写入 `docs/specs/YYYY-MM-DD--{feature}.md`
4. **批准摘要**：构建/不构建/方法/关键决策/未知项
5. **关键决策日志**：3-5 个决策（选项+理由）
