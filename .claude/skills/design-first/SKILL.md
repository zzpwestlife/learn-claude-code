## 铁律

设计先于实施。没有代码、脚手架或伪代码，直到用户批准设计。
即使"简单"的改动也需要设计——简短即可，但必须呈现并获得批准。

---

## 总览：复杂度驱动的设计流程

所有请求先经过**阶段1定级**，然后按级别走不同深度的流程：

```
用户请求
  ↓
阶段1 ─ 探索上下文 + 定级
  ↓
┌─────────┬───────────┬──────────────┐
│  轻量 L │  中等 M   │   重度 H     │
├─────────┼───────────┼──────────────┤
│ 1问     │ ≤3问      │ ≤5问/自检    │ ← 阶段2 追问
│ 跳过    │ 2角度     │ 4角度全做    │ ← 阶段3 攻击
│ 1段合并 │ 正常精简  │ 逐段审批     │ ← 阶段4 设计
│ 跳过    │ 精简      │ 完整+Git     │ ← 阶段5 文档
├─────────┴───────────┴──────────────┤
│       阶段6 ─ 批准摘要 → 停止       │
└─────────────────────────────────────┘
```

### 定级标准

| 级别 | 条件 | 典型场景 |
|------|------|---------|
| **L 轻量** | ≤3文件、无新依赖、无schema/API变更 | CSS调整、配置修改、单函数工具 |
| **M 中等** | 4-8文件、或已有依赖新用法、或数据迁移 | 加缓存层、新API端点、重构模块 |
| **H 重度** | >8文件、新系统/服务、新语言/运行时、多团队 | 认证系统、微服务拆分、平台级功能 |

**边界模糊时**：文件数少但引入新运行时→按M。文件数4-8但改动都是同类重复→按L。仍然存疑→按高一级处理。

---

## 阶段1：探索与定级

**目的**：了解现状，确定复杂度级别。

### 步骤

1. **确认工作路径**：`pwd` 或 `git rev-parse --show-toplevel`（不要假设路径）
2. **探索项目结构**：
   - 入口文件和配置（`package.json`/`pyproject.toml`/`go.mod`）
   - 与需求相关的模块（搜索关键词定位已有实现）
   - 最近提交：`git log --oneline -10`
   - L级可跳过：已有测试模式、ADR/设计文档
3. **评估范围**：
   - 多个独立子系统→先分解，每个子项目走独立设计循环
   - 单一可管理范围→定级后继续
4. **输出定级结论**：声明级别（L/M/H）和判定理由

### "直接做"处理

用户说"直接做"或等效表达时：
1. 仍完成本阶段的定级
2. L级：视为批准推荐方案，快速通道直达阶段6
3. M/H级：告知"这个需求比看起来复杂"，说明原因，征求是否仍想快速推进
4. 无论哪级：必须声明选择了哪个方案及理由。不在 `/think` 内实施。

### 🚪 门控

> 声明了级别（L/M/H）并告知用户 → 进入阶段2

---

## 阶段2：深度追问

**目的**：收集足够信息以自信完成阶段4的全部设计section。

### 🥷 标记开始

首次提问前行内标记：🥷 让我们深入探索这个想法。

### 规则

- **一次一问**，不堆叠多个问题
- **多选优先**，比开放式问题更易回答
- **每问给推荐答案**，帮用户快速决策
- **代码库优先**：能通过探索代码回答的不问用户

### 追问顺序（默认优先级，上游已确定时跳过下游）

| 优先级 | 焦点 | 示例问题 |
|--------|------|---------|
| 1 | **目的** | "这个缓存是降DB负载，还是降用户感知延迟？" |
| 2 | **成功标准** | "P95延迟从500ms降到100ms算成功？" |
| 3 | **约束** | "需要兼容现有Redis 5.x集群吗？" |
| 4 | **风险容忍** | "缓存失效导致5分钟数据不一致可接受吗？" |
| 5 | **利益相关者** | "移动端和Web端行为需要一致吗？" |

### 问题上限与终止

| L | M | H |
|---|---|---|
| 1问 | ≤3问 | 无上限，每5问自检："能填满阶段4所有section了吗？" |

**终止条件**（任一满足即停止追问）：
- 能自信填写阶段4所有设计section（架构、组件、数据流、错误处理、测试）
- 达到级别问题上限
- 用户说"够了"/"继续下一步"
- 剩余不确定点可通过探索代码库解决

### 🚪 门控

> 声明"信息充足，进入方案探索" → 进入阶段3

---

## 阶段3：方案探索与攻击测试

**目的**：找到最优方案，压力测试其弱点。

### L级：跳过本阶段（风险已在定级过滤）→ 直接进入阶段4

### M/H级：

#### Step 1：检查官方方案

按优先级搜索：框架内置 → 官方推荐模式 → 生态标准库

- 查询方式（按优先级）：Context7 MCP（如已安装）→ WebSearch 搜索官方文档 → 直接读项目内 node_modules/vendor 中的库源码
- **官方方案存在时必须作为选项1**，推荐自定义需解释理由
- 未经明确批准，禁止添加新语言或运行时

#### Step 2：提出2-3个方案

每个方案包含：一句话总结 · 工作量估算 · 风险 · 依赖的现有代码。**始终包含一个最小化选项。**

#### Step 3：攻击测试

| 攻击角度 | 核心问题 |
|---------|---------|
| **依赖失败** | 外部服务宕机时能优雅降级吗？ |
| **规模爆炸** | 10倍负载下哪步先崩？ |
| **回滚成本** | 发布后发现方向错，回滚有多难？ |
| **过早优化** | 在构建不立即需要的功能吗？ |

**深度适配**：M级→2个最相关角度，每个1-2句 ｜ H级→4个全做，每个1段+具体数据/场景

**处理结果**：
- 攻击成立→变形设计并呈现变形版本
- 完全推翻→放弃该方案并告知理由
- **未披露攻击弱点的设计不得呈现**
- **用户拒绝所有方案**→追问"哪些约束未满足？"，带着新约束重新生成方案。连续两轮仍被拒绝→建议缩小范围或分解问题后重新进入阶段1

### 🚪 门控

> 推荐方案确定，所有攻击结果已向用户披露 → 进入阶段4

---

## 阶段4：分段设计与验证

**目的**：逐段呈现设计，每段获取用户批准。

### 段落规格

| L | M | H |
|---|---|---|
| 合并为1段（~100字） | 3-4段（每段~150字） | 5段各自审批（每段≤300字） |

### 5个设计Section

> H级全做并逐段审批，M级合并相近项为3-4段，L级合为1段总述。

| Section | 回答什么 |
|---------|---------|
| **架构** | 系统如何组织？模块划分？ |
| **组件** | 主要构建块及职责，接口定义 |
| **数据流** | 数据从哪来→经过什么→到哪去（>3组件交换数据时画ASCII图并检查循环） |
| **错误处理** | 失败场景及应对，降级策略 |
| **测试** | happy path · error path · edge case，具体列出每条 |

### 每段门控

> "这部分设计合理吗？有需要调整的吗？"
> 用户批准→下一段 ｜ 拒绝→问具体偏离点，修改后重新呈现本段

### 验证清单（所有段呈现完后执行）

| 检查项 | 阈值 | 操作 |
|--------|------|------|
| 文件+服务数 | >8文件或>1新服务 | 明确标注 |
| 数据交换复杂度 | >3组件交换数据 | ASCII图+检查循环依赖 |
| 测试覆盖 | 每条有意义路径 | happy/error/edge全列出 |
| 回滚能力 | — | 确认：能否不回滚数据地回滚？ |
| 凭证依赖 | — | 列出每个API key/token/第三方账户+动机 |
| 外部依赖 | — | 列出每个MCP/API/CLI，验证可达性 |
| **占位符检查** | 0个 | 搜索TBD/TODO/"later implement"/"to be determined"并消除 |

### 🚪 门控

> 验证清单全部通过 → 进入阶段5

---

## 阶段5：设计文档与自检

**目的**：将设计固化为可追溯的文档。

### L级：跳过本阶段（阶段4的1段设计即为文档）→ 直接进入阶段6

### M/H级：

#### 写入Spec文件

默认路径：`docs/specs/YYYY-MM-DD--{feature}.md`（用户偏好覆盖此默认值）

```markdown
# 设计：[功能名称]
## 概述（1-2段：是什么+为什么重要）
## 范围
### 构建（明确列出包含内容）
### 不构建（明确out-of-scope列表）
## 架构
## 组件
## 数据流
## 错误处理
## 测试策略
## 关键决策（每个决策：选项+理由）
## 未知项（显式延期：原因+负责人。不是模糊的差距）
```

#### 自检四问

| # | 问题 | 动作 |
|---|------|------|
| 1 | 有TBD/TODO/模糊需求？ | 就地修复 |
| 2 | 任何部分自相矛盾？ | 就地修复 |
| 3 | 范围足够聚焦到单一实施计划？ | 需要则分解 |
| 4 | 有需求可被两种方式解读？ | 选一个并明确 |

#### Git提交

将spec文件提交到git。

### 🚪 门控

> "Spec已写入并提交。请查看，如需修改请告知。"
> 用户批准→阶段6 ｜ 用户要求修改→修改后重跑自检四问

---

## 阶段6：批准摘要

**目的**：输出最终设计摘要，等待实施指令。所有级别都执行此阶段。

```
═══════════════════════════════════════
         批准的设计摘要
═══════════════════════════════════════
## 构建：这是什么（1段）
## 不构建：明确的out-of-scope列表
## 方法：选择的方案 + 理由
## 关键决策：3-5个决策 + 理由
## 未知项：显式延期 + 原因 + 负责人
═══════════════════════════════════════
```

**输出后停止。** 实施只在用户明确请求时开始。

## 全局回退路径

| 触发 | 行为 |
|------|------|
| 用户说"不对/重来" | 问具体偏离点，回退到最近相关阶段，已确认部分保留，不从零开始 |
| 技术探索发现方案不可行 | 回到阶段3，排除该方案，从剩余方案中选择 |
| 用户中途改变需求 | 回到阶段1重新定级，已确认部分保留 |

## 防御性规则

| 场景 | 规则 |
|------|------|
| 不确定工作路径 | 第一个文件操作前 `pwd` |
| 计划依赖外部工具/MCP | 交接前验证工具可用性，不要到实施中途才发现 |
| 准备添加新语言/运行时 | 未经用户明确批准，禁止 |
| 外部API有区域差异 | 编码前列出所有区域差异 |
| 计划中出现占位符 | **有占位符的计划是对将来做计划的承诺。** 批准前必须全部消除 |
