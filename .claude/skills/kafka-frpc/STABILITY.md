# kafka-frpc Skill 稳定性问题记录

本文记录 skill 调试过程中反复出现的问题及其根本原因和解决方案，供后续优化参考。

---

## 问题 1：地区选项只显示 4 个

**现象**：AskUserQuestion 问地区时只出现 AU / CA / HK / HKVA + "Type something"，其余地区看不到。

**根本原因**：AskUserQuestion 工具硬限制最多 4 个 options，第 5 个自动变成 "Type something"。10 个地区根本无法全部展示。

**解决方案**：地区改为纯文本追问（不调用任何工具），在回复文本中列出所有地区代码。

---

## 问题 2：地区选项变成多选（复选框）

**现象**：地区选择界面出现 `[ ]` 复选框，用户可以勾选多个地区。

**根本原因**：Claude 默认行为倾向使用 multiSelect，文字禁令（"单选"）无效。

**解决方案**：彻底弃用 AskUserQuestion 来问地区，改为纯文本追问。文本回复没有 UI 组件，无法出现复选框。

---

## 问题 3：Claude 自行添加"全部地区"选项

**现象**：选项里出现"全部地区（HK/AU/HKVA/US 等）""其他/多个地区"之类的批量选项。

**根本原因**：Claude 推断"用户可能想接多个地区"并自行发挥，即使 skill 没有这个选项。

**解决方案**：在 Step 1 最顶部（所有指令之前）加硬约束："一次只接入一个地区，无论用户如何表达都不批量处理。"把约束放在流程描述之前，比放在中间有效。

---

## 问题 4：多个问题被合并成一次 AskUserQuestion

**现象**：界面顶部出现多个 tab（地区 / 用途 / 配置文件），Claude 把所有问题塞进一次调用。

**根本原因**：AskUserQuestion 支持多 question 参数，Claude 倾向于"一次问完"减少交互轮次。

**解决方案**：
- 地区改为文本追问（物理上无法合并进 AskUserQuestion）
- 其他问题明确写"每次只问一个，不合并"
- 第四步的文本追问也标注"直接在回复文本中问，不调用工具"

---

## 问题 5：配置文件选择出现地区专属路径

**现象**：Claude 把 `conf/hk/conf.toml`（推荐）放在选项第一位，`conf/conf.toml` 排第二。

**根本原因**：Step 0 读了项目结构，发现 `conf/hk/conf.toml` 存在，Claude 推断"HK 配置应该写地区专属文件"并覆盖 skill 指令。

**解决方案**：
- 在 Step 0 末尾加明确规则："即使发现地区专属文件，写入目标仍默认为 `conf/conf.toml`，不得擅自改为地区专属路径。"
- 把配置文件确认问题改为 yes/no 二选一："配置将写入 conf/conf.toml，确认吗？"——不给 Claude 机会自由发挥第三个选项。

---

## 问题 6：配置文件被问两次

**现象**：Step 1 采集时问了一遍配置文件，Step 2 写入前又问了一遍。

**根本原因**：skill 在两个步骤都有配置文件确认逻辑，Claude 忠实执行了两次。

**解决方案**：只在 Step 2（写入前）问一次，Step 1 采集阶段完全删除配置文件相关问题。写入后不再重复确认。

---

## 问题 7：使用环境问题被跳过，直接用线上地址（第一轮修复）

**现象**：Claude 未问使用环境，直接在配置里写了 `address = "fns://kafka_finrd_mq"`（线上地址）。

**根本原因**：
1. 环境问题排在第三步，Claude 在有足够上下文时倾向于跳过"多余"的问题
2. Section 1 所有配置模板的默认 address 值是 `fns://`，Claude 复制模板时直接用，没有看注释

**解决方案（第一轮，仍然不够）**：
1. 第三步加"**必须**询问，不得跳过"，并明确"默认为本地/测试"
2. 把所有配置模板中的 address 默认值从 `fns://` 改为 `<broker-ip>:<port>`，`fns://` 降为注释

---

## 问题 8：环境问题仍被跳过，模板改动无效（第二轮）

**现象**：改了模板地址为 `<broker-ip>:<port>` 后，Claude 依然写出 `fns://kafka_finrd_mq`。

**根本原因**：
- Claude **不从模板复制**，而是从训练知识里直接生成配置值。它知道 frpc kafka 的线上地址是 `fns://kafka_finrd_mq`，所以绕过了模板
- 环境问题放在第三步，Claude 在前两步已收集到足够信息，直接跳到生成配置

**解决方案（双重）**：
1. **把环境问题提到 Step 1 第一步**（最先执行），成为所有后续步骤的前提，物理上无法跳过
2. **模板 address 改为 `❌必须替换：本地填 IP:port，线上填 fns://kafka_finrd_mq`**：❌ 符号让 Claude 无法把它当作有效值使用，必须替换为真实内容

## 核心规律总结

| 问题类型 | 无效的修复方式 | 有效的修复方式 |
|---------|-------------|-------------|
| Claude 用错 UI 组件（如复选框） | 加"禁止"文字 | 改变问法使该 UI 组件物理上不可出现（改为文本追问） |
| Claude 自行发挥选项内容/顺序 | 加"不得增减""推荐第一位"文字 | 改为 yes/no 二选一，彻底消除自由发挥空间 |
| Claude 用项目上下文覆盖 skill 指令 | 加强调词（最高优先级、不可违反） | 在探查步骤后紧跟明确规则，说明探查结果不影响该决策 |
| Claude 跳过某个问题 | 加"必须""不得跳过"文字 | 把该问题提到最前面成为第一步，或改模板默认值使跳过也能得到安全结果 |
| Claude 绕过模板直接生成已知值 | 改模板默认值（Claude 不从模板读） | 用明显错误的占位符（如 ❌ 前缀）迫使 Claude 必须替换 |
| 多值合并追问导致解析失败 | 加"逐一追问"文字 | 改为每次只问一个问题，等待回复后再继续 |
| 选项顺序被 Claude 自行排列 | 写"第一个选项是 X" | 彻底删除该问题，或将期望结果硬编码为唯一选择 |

## 问题 9：在目标服务里修改无效，skill 改了等于没改

**现象**：所有修复都做了，目标服务里 address 仍然是 `fns://kafka_finrd_mq`。

**根本原因**：`kafka-frpc` skill 存在两份独立的文件：
- **本地项目版**：`learn-claude-code/.claude/skills/kafka-frpc/SKILL.md`（只在 learn-claude-code 项目内生效）
- **全局版**：`~/.claude/skills/kafka-frpc/SKILL.md`（其他所有项目使用这份）

我们一直在改本地版，目标服务加载的是全局版，两者完全隔离，互不影响。

**解决方案**：改完本地版后，手动同步到全局路径：
```bash
cp .claude/skills/kafka-frpc/SKILL.md ~/.claude/skills/kafka-frpc/SKILL.md
```

**预防措施**：修改 skill 后，**先确认在目标环境里哪份文件实际生效**，再决定改哪个路径。

## 设计原则

1. **结构优于文字**：改变问法的结构（文本 vs 工具调用），比增加文字禁令更可靠
2. **位置决定优先级**：最重要的问题放第一步，Claude 不会跳过它
3. **模板要不可误读**：模板中不应出现任何看起来像真实值的占位符，用 ❌/FIXME/TODO 等明显标记
4. **问题越少越稳定**：能默认的就默认，减少需要 Claude 决策的环节
5. **上下文是双刃剑**：Step 0 探查项目结构会给 Claude 提供"推断依据"，必须在探查步骤后立即声明哪些推断被禁止
