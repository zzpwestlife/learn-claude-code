# 心理模型应用检查清单 (Mental Model Application Checklist)

此清单旨在确保在开发流程的各个阶段，系统性地应用核心心理模型（奥卡姆剃刀、费曼技巧等），以提升架构质量和代码可维护性。

## 1. 意图定义阶段 (Intent Definition Phase) - **费曼技巧 (Feynman Technique)**

- [ ] **简单语言描述**: 我能否用非技术人员也能听懂的简单语言描述用户故事？(拒绝专业术语堆砌)
- [ ] **比喻与类比**: 是否使用了恰当的比喻或类比来解释复杂概念？
- [ ] **因果清晰**: `spec.md` 中的需求是否清晰地阐述了"为什么"（Why）而不仅仅是"做什么"（What）？
- [ ] **教学视角**: 我是否将 Spec 视为给 AI（大一新生）的教学讲义？如果 AI 生成错误，我是否反思了 Spec 的清晰度？

## 2. 技术规划阶段 (Technical Planning Phase) - **奥卡姆剃刀 (Occam's Razor)**

- [ ] **存在性测试**: 新增的 Agent/Skill/依赖库是否绝对必要？能否通过优化 Prompt 或组合现有能力实现？
- [ ] **依赖最小化**: 引入的每一个外部库，是否都能证明其不可替代性？（能否用标准库在 50 行内解决？）
- [ ] **YAGNI 审查**: 计划中是否包含"未来可能需要"的功能？如果有，请剔除。
- [ ] **实体精简**: 架构设计中是否引入了多余的层级或抽象？

## 3. 任务分解阶段 (Task Decomposition Phase) - **盖尔定律 (Gall's Law)** & **帕累托法则 (Pareto Principle)**

- [ ] **MVP 优先**: 第一个任务（Task 1）完成后，系统是否能运行并演示核心功能？（盖尔定律）
- [ ] **核心价值聚焦**: 哪 20% 的功能构成了 80% 的价值？是否优先安排了这部分任务？（帕累托法则）
- [ ] **渐进式演进**: 任务是否按照从简单到复杂的顺序排列，避免"大爆炸"式开发？

## 4. 自动化实现与审查阶段 (Implementation & Review Phase) - **第一性原理 (First Principles)** & **二阶思维 (Second-Order Thinking)**

- [ ] **代码自解释**: 变量名和函数名是否清晰地表达了意图（Why）？（费曼技巧在代码中的应用）
- [ ] **无隐含假设**: 代码实现是否直接反映了 Spec 的要求，没有依赖未声明的假设？（第一性原理）
- [ ] **长期维护评估**: 引入的临时解决方案或"黑科技"，在 2 年后是否会成为维护噩梦？（二阶思维）
- [ ] **死代码清理**: 是否定期运行了死代码检测，并果断删除了未引用的代码？（奥卡姆剃刀）

---

**使用说明**:
在每次进行 `/plan` 或 `/review` 时，请参考此清单进行自我审查。
