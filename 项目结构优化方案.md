# 项目结构优化方案

基于对《项目结构说明书》的深入分析，本方案旨在优化项目结构，提升代码库的可维护性、安全性和业务价值。

## 1. 项目概况与评估摘要

本项目主要是一个关于 Claude Code 的综合学习资源库，包含了大量的配置示例、教程文档 (`docs/course`) 以及实用的工具脚本 (`.claude/skills`)。项目内容丰富，涵盖了从基础配置到高级 Agent 开发的全方位指南。

**评估摘要：**
*   **业务价值**：**高**。提供了系统的 Claude Code 学习路径和实用的自动化工具，显著降低了用户的学习成本和使用门槛。
*   **技术债务**：**中**。目录结构在部分区域（如 `.claude` 内部）过于复杂，存在冗余文件和临时文件残留，依赖管理较为分散。
*   **安全合规**：**中**。需重点关注敏感配置文件的管理和第三方脚本的安全性。

## 2. 多维度详细评估

### 2.1 业务价值评估
*   **核心资产**：`docs/course` 下的教程文档是项目的核心资产，具有极高的教育价值。
*   **工具赋能**：`.claude/skills` 下的工具集（如 `union-search-skill`, `auto-doc`）直接增强了 Claude 的能力，属于高价值的功能扩展。
*   **最佳实践**：`.claude/agents` 和 `.claude/commands` 沉淀了大量 Prompt Engineering 的最佳实践，可复用性强。

### 2.2 技术债务评估
*   **结构复杂度**：`.claude` 目录承担了过多的职责（配置、文档、脚本、日志、临时文件），导致结构臃肿，维护困难。
*   **冗余文件**：存在多个 `README.md` 和 `CLAUDE.md` 分散在不同子目录，部分内容可能重复，增加了文档维护成本。
*   **临时文件残留**：`.claude/audit`, `.claude/tmp`, `.claude/logs` 等目录包含了大量的过程数据，非项目源码，不应长期保留在版本控制中。
*   **依赖管理分散**：Python (`requirements.txt`) 和 Node.js (`package.json`) 的依赖分散在各个 Skill 子目录中，缺乏统一的版本控制和升级机制。

### 2.3 安全合规评估
*   **敏感信息泄露风险**：`.env` 文件（如 `.claude/skills/union-search-skill/.env`）若未被 `.gitignore` 正确忽略，存在泄露 API Key 的风险。
*   **脚本执行权限**：项目包含大量 Shell 和 Python 脚本，需确保其来源可信，并限制非必要的执行权限。
*   **第三方组件安全**：引用的外部工具（如 `yt-dlp`）和 Python/Node 依赖包需定期扫描漏洞。

## 3. 优化建议分类

### 3.1 保留 (Retain)
*   **核心配置**：`.claude/AGENTS.md`, `.claude/settings.json`, `.claude/claudeignore`。
*   **Agent 定义**：`.claude/agents/*.md`。
*   **自定义命令**：`.claude/commands/*.md`。
*   **核心文档**：`docs/course`, `docs/guides`, `docs/best_practices`, `README.md`, `CLAUDE.md`。
*   **核心技能**：经过验证且频繁使用的 Skills，如 `union-search-skill`（建议重构）, `auto-doc`。
*   **架构文档**：`ARCHITECTURE.md`, `SOUL.md`。

### 3.2 重构 (Refactor)
*   **`.claude/lib`**：该目录下的脚本（如 `architect.py`, `audit_skills.py`）较为零散。建议分类整理到 `.claude/scripts` 或具体的 Skill 目录中，保持功能的内聚性。
*   **`.claude/skills/union-search-skill`**：该模块体积庞大且包含独立的环境配置。建议将其标准化为独立的子模块，或在项目根目录下提供统一的依赖安装脚本。
*   **文档结构**：整合分散的 `CLAUDE.md` 文件，确保单一事实来源（Single Source of Truth），避免信息冲突。

### 3.3 迁移 (Migrate)
*   **归档文档**：将 `docs/archive` 和 `.claude/docs/archive` 中的内容迁移到独立的 git 分支（如 `archive` 分支）或专门的归档仓库，以减小主仓库体积。
*   **大型资源**：若 `docs/course/assets/images` 中的图片体积持续增长，建议迁移到对象存储（OSS/S3）或使用 Git LFS 进行管理。

### 3.4 删除 (Delete)
根据《项目结构说明书》中的标记，建议删除以下文件/目录：
*   **`.claude/audit`**：审计日志，属于运行时数据。
*   **`.claude/logs`**：运行日志，属于运行时数据。
*   **`.claude/tmp`**：临时文件，属于运行时数据。
*   **`.claude/docs/archive`**：已确认过时且无参考价值的文档。
*   **`.claude/lib` 中未被引用的临时脚本**：如 `check-complete.sh` 等若确认无用则删除。
*   **`.trae`**：如果项目主要面向 Claude Code 用户，且 Trae 配置非必须，可考虑删除或添加到 `.gitignore`。

## 4. 实施计划

### 第一阶段：清理与瘦身 (Clean & Slim)
**目标**：移除无用文件，减小项目体积，消除噪音。
1.  **执行清理命令**：
    ```bash
    rm -rf .claude/audit
    rm -rf .claude/logs
    rm -rf .claude/tmp
    rm -rf .claude/docs/archive
    ```
2.  **更新 `.gitignore`**：确保上述目录被忽略，防止再次提交。

### 第二阶段：结构优化 (Refactor Structure)
**目标**：提升代码组织的逻辑性和可维护性。
1.  **整理 `.claude/lib`**：分析其中脚本的用途，有用的移动到 `.claude/scripts` 并重命名为更具描述性的名称，无用的直接删除。
2.  **标准化 Skill 结构**：确保每个 Skill 都有清晰的 `README.md` 和依赖说明，尽量统一依赖管理方式。

### 第三阶段：文档与规范 (Documentation & Standards)
**目标**：统一文档入口，规范开发流程。
1.  **统一入口**：确保根目录的 `README.md` 和 `CLAUDE.md` 能索引到所有关键文档。
2.  **依赖审查**：扫描所有 `requirements.txt` 和 `package.json`，更新过时依赖，修复安全漏洞。
3.  **敏感信息检查**：使用工具（如 `git-secrets`）扫描历史提交中的敏感信息。

### 第四阶段：自动化与维护 (Automation & Maintenance)
**目标**：建立长期的维护机制。
1.  **自动化清理脚本**：编写脚本定期清理 `.claude/logs` 等临时文件。
2.  **CI/CD 集成**：如果可能，配置 CI 流程自动检查文档链接有效性和代码规范。
